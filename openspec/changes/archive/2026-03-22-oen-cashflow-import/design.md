## Context

看我笑話工作室的財務平台目前透過手動輸入方式記錄收支。實際金流來自應援（Oen）平台，每月產出「撥款明細」xlsx（所有金流彙總）及多份「活動報名狀態」xlsx（各場活動票券明細）。目前需要人工交叉比對才能知道每筆金流對應哪場活動，再手動輸入到平台。

現有技術架構：前端 vanilla JS + HTML，後端 Google Apps Script，資料存於 Google Sheets。前端透過 `js/api.js` 呼叫 GAS API，使用 `action` 參數路由。

## Goals / Non-Goals

**Goals:**

- 用戶上傳 xlsx 後，自動解析並分類所有金流來源
- 清楚顯示已上傳的檔案狀態，避免遺漏
- 提供配對結果儀表板，一目了然每場活動的收入、手續費
- 一鍵將配對結果批次寫入 Google Sheets 現有收支紀錄

**Non-Goals:**

- 不處理應援以外的金流來源
- 不自動排程匯入（用戶手動觸發）
- 不修改現有收支紀錄頁面或分析頁面的邏輯
- 不處理退款沖銷（退款紀錄僅顯示在儀表板，不寫入 Sheets）

## Decisions

### XLSX 前端解析（SheetJS CDN）

使用 SheetJS（xlsx.js）在瀏覽器端解析 xlsx 檔案，不需要上傳到伺服器。

- 替代方案：在 GAS 端解析 → 需要先上傳到 Google Drive，流程複雜且慢
- 選擇理由：前端解析即時回饋，不需改動太多後端，與現有架構一致

### 三層配對策略

配對金流明細與活動報名的邏輯：

1. **定期定額自動分類**：金流類型為「定期定額會費」→ 直接歸類為「付費會員」
2. **付款時間 + 姓名配對**：使用金流明細的「付款時間」與活動報名的「報名日期」，5 分鐘內 + 同姓名 → 配對成功
3. **Email fallback**：未配對的用「電子郵件」=「購買人 Email」再試一次（處理超商付款等時間差大的情況）
4. **手動指定**：仍未配對的標記為「未配對」，用戶可手動選擇所屬活動

- 替代方案：只用姓名 + 金額 → 同名同額多筆會誤配
- 選擇理由：付款時間是最精確的配對鍵，Email 作為 fallback 處理超商付款的時間差問題

### 活動名稱從檔名擷取

活動報名 xlsx 的檔名格式為 `日期_活動名稱_活動報名狀態_N筆.xlsx`，系統從檔名自動擷取活動名稱。

- 替代方案：讓用戶手動輸入活動名稱 → 增加操作步驟
- 選擇理由：應援平台的檔名格式穩定，自動擷取減少操作負擔

### 活動名稱對應演出清單

從檔名擷取的活動名稱（如「看我笑話｜第 2 季 4 月號」）需對應到 Google Sheets 中的演出名稱（如「看我笑話 4 月號」）。系統提供下拉選單讓用戶確認或修改對應關係。

### 批次寫入 API

新增 GAS action `batchImportTransactions`，接受交易陣列一次寫入多筆。每筆包含：演出名稱、分類、備註、金額（含稅收取金額）、日期。手續費另外作為「平台手續」支出寫入。

- 替代方案：逐筆呼叫 `addTransaction` → 220 筆 = 220 次 API call，太慢
- 選擇理由：批次寫入一次完成，減少 API call 次數

### 匯入結果分組方式

將金流依據以下邏輯分組寫入收支紀錄：

- **演出票房**（按活動）：每場活動的票券收入彙總為一筆，備註含票種明細
- **付費會員**：所有定期定額會費彙總為一筆，備註含方案明細（149 元 x N 人、1699 元 x N 人）
- **平台手續**（按活動）：每場活動的手續費彙總為一筆支出
- **付費會員手續費**：會員費的手續費彙總為一筆支出

## Risks / Trade-offs

- [風險] 應援平台改變 xlsx 欄位名稱 → 解析邏輯需要更新。緩解：欄位名稱定義集中管理，改動範圍小
- [風險] 同一人同時間買兩場不同活動的票 → 時間配對可能誤配。緩解：實務上極少發生，未配對項目可手動調整
- [取捨] 前端解析 xlsx 需要載入 SheetJS（~300KB） → 只在匯入頁載入，不影響其他頁面效能
