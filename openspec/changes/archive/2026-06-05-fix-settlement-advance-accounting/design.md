## Context

目前結算流程（`addSettlement`）將整筆匯款金額存入 `已收款淨利`，並自動把該成員所有未結清代墊標為已結清。問題在於：實際匯款中若包含代墊還款，代墊部分不屬於利潤分配，不應計入 `settled`。

受影響邏輯（`js/analytics.js`）：

```
unsettledNet = annualNet - settled       // 未收款淨利
toPay        = unsettledNet + advances   // 需匯款金額
```

當代墊被結清但 `settled` 沒有扣除代墊金額，`unsettledNet` 就被低估，`toPay` 顯示的金額比實際少。

## Goals / Non-Goals

**Goals:**

- 結算時，settlement 僅記錄「利潤結算」部分（= 總匯款 - 代墊還款）
- Modal 顯示代墊金額並讓使用者決定是否一起結清
- 提供歷史資料修正工具，修正過去錯誤記錄的 settlement 金額

**Non-Goals:**

- 不支援部分代墊結清（一律全清）
- 不拆分 settlement 紀錄為兩筆（利潤 + 代墊）
- 不修改 transactions 的資料結構

## Decisions

### 結算金額語意：儲存利潤部分，不含代墊還款

**決定**：`addSettlement` 接收的 `amount` 為「利潤結算金額」（已扣除代墊），前端在 submit 前自動計算並傳入扣除後的值。

**替代方案**：傳入總匯款，後端自動扣除未結清代墊再存入。後端方案需要 GAS 查詢未結清代墊，增加複雜度；前端拆分更直覺，且 modal 可即時呈現拆分細節給使用者確認。

### Modal UI：動態顯示代墊區塊

選擇成員後，若有未結清代墊：

```
代墊未結清：$5,000
☑ 一起結清代墊

總匯款金額：[20,000]
─────────────────────
利潤結算：$15,000 / 代墊還款：$5,000
```

金額欄位語意從「結算金額」改為「總匯款金額」，勾選後即時計算並顯示拆分。

**替代方案**：讓使用者直接輸入「利潤結算金額」（不含代墊），操作更少但容易搞錯——使用者習慣思考的是「我匯了多少錢」。

### 歷史資料修正：GAS 修正函式 + 前端觸發

**問題**：過去已結清的代墊（`t.settled = true`），但 settlement 紀錄沒有扣除。

**修正策略**：
1. GAS `fixAdvanceSettlements()` 掃描每位成員已結清的代墊總額
2. 比對該成員的 settlement 紀錄，找出金額最大的一筆進行修正（減去代墊總額）
3. 若修正後金額 ≤ 0，則改為修正第二大的紀錄，以此類推
4. 前端 analytics 頁新增「修正代墊帳」按鈕，顯示預覽後由使用者確認執行

**前提假設**：過去的 $20,000 結算金額中，代墊部分已涵蓋在內（使用者已確認）。

## Risks / Trade-offs

- **修正工具假設代墊已被包含在 settlement 內**：若實際情況不同，修正方向會錯。已透過使用者確認解決。
- **修正後 `unsettledNet` 增加**：使用者需要實際匯款更多才能完成分配，這是正確行為，但數字變化可能令人疑惑——modal 應明確說明修正意義。
