## Context

看我笑話工作室是 8 人日式漫才喜劇團體，目前用陽春 Google Sheets 手動記帳。本次要建立一個三頁式線上平台（收支紀錄、演出準備、財務分析），前端為純 HTML + JS 靜態網頁，部署於 GitHub Pages，資料透過 Google Apps Script API 存取 Google Sheets。

團員：傑哥、柏文、巧達、芭樂、又又、兔子、大弋、竹節蟲（＋「其他」可自填）。

無登入驗證機制，以下拉選人方式操作。

## Goals / Non-Goals

**Goals:**

- 提供簡潔好用的收支登記介面，取代手動編輯 Google Sheets
- 每檔線下演出可獨立追蹤損益
- 追蹤墊款與結清狀態
- 提供月/年淨利、收入佔比、支出佔比等財務報表
- 提供每月演出準備 Checklist，從固定模板生成，支援進度追蹤
- 演出清單跨頁面共用
- MUJI 簡潔高質感清爽視覺風格

**Non-Goals:**

- 不處理登入驗證或權限控管
- 不追蹤線上影片帳務
- 不追蹤周邊商品帳務
- 不做即時多人協作衝突處理（依賴 Google Sheets 本身的併發處理）
- 不做行動裝置 App（但網頁須具備基本 RWD）

## Decisions

### Google Sheets 資料表結構

採用四張工作表：

| 工作表 | 用途 | 主要欄位 |
|--------|------|---------|
| `演出清單` | 所有演出檔次 | 演出名稱、建立日期、狀態 |
| `收支紀錄` | 所有收支項目 | 演出名稱、項目、金額（正=收入/負=支出）、墊款人、結清狀態、日期、登記人 |
| `Checklist` | 待辦進度 | 演出名稱、類別、項目名稱、負責人、進度（未開始/進行中/已完成）、備註 |
| `Checklist模板` | 固定模板定義 | 類別、項目名稱、預設負責人 |

**理由**：扁平化結構最適合 Google Sheets，用「演出名稱」作為各表之間的關聯鍵，查詢與篩選都簡單。

### 前端技術選擇

純 HTML + CSS + vanilla JS，不使用任何框架。

**理由**：團隊規模小、頁面僅三頁、互動邏輯不複雜，vanilla JS 足以應付，且部署在 GitHub Pages 零依賴。

### 前端頁面結構

三個獨立 HTML 頁面，共用一份 CSS 和共用 JS 模組：

```
index.html          ← 收支紀錄（預設首頁）
checklist.html      ← 演出準備
analytics.html      ← 財務分析
css/style.css       ← 共用樣式（MUJI 風格）
js/api.js           ← Google Apps Script API 呼叫封裝
js/shared.js        ← 共用元件（導覽列、演出下拉選單、團員選單）
js/transaction.js   ← 收支紀錄頁邏輯
js/checklist.js     ← 演出準備頁邏輯
js/analytics.js     ← 財務分析頁邏輯
```

### Google Apps Script API 設計

部署為 Web App，以 GET/POST 搭配 `action` 參數路由：

| action | 方法 | 用途 |
|--------|------|------|
| `getShows` | GET | 取得演出清單 |
| `addShow` | POST | 新增演出 |
| `getTransactions` | GET | 取得收支紀錄（可依演出篩選） |
| `addTransaction` | POST | 新增收支 |
| `updateTransaction` | POST | 更新收支（結清狀態） |
| `getChecklist` | GET | 取得某演出的 Checklist |
| `initChecklist` | POST | 從模板初始化某演出的 Checklist |
| `updateChecklistItem` | POST | 更新 Checklist 項目進度/負責人 |
| `addChecklistItem` | POST | 新增自訂 Checklist 項目 |

**理由**：Google Apps Script Web App 免費、無需額外伺服器、直接操作 Sheets，對此規模最經濟實用。

## Risks / Trade-offs

- **[Google Apps Script 冷啟動延遲]** → 首次載入可能需 1-2 秒，加入 loading 狀態提示
- **[無驗證機制]** → 任何人知道網址都能操作。風險可接受，因為僅團隊內部使用，且資料非高度機密
- **[併發寫入衝突]** → 兩人同時操作可能產生衝突。風險低，因為團隊僅 8 人，同時操作機率小
- **[Google Sheets 效能上限]** → 當資料量成長到數千筆以上可能變慢。短期內不會達到此限制
- **[API Key 暴露]** → Apps Script Web App URL 會暴露在前端 JS 中。風險可接受，因為無敏感資料且無驗證需求
