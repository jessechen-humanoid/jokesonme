## Context

現有系統的共同基金是「虛擬提撥」：20% × 共同淨利，單純展示用，沒有「餘額」語義。現在要升級成「有餘額的虛擬帳戶」，且要同時引入「稅務預留」自動化機制。

兩件事在資料模型上有交集：
- 稅務預留支出會進入共同支出 / 非共同分配的標準計算流程（沿用 `excludedMembers` 語義）
- `paidByFund` 支出則要從共同支出中完全隔離

清楚切開兩種語義，才能讓成員年度分配淨利 / 共同基金餘額兩個數字都正確。

**關鍵既有資料模型**：
- 交易欄位（Google Sheet「收支紀錄」）：date, show, category, notes, amount, recorder, excludedMembers, advancePayer, settled
- 共同交易定義：`excludedMembers === ''`
- 成員年度分配淨利 = (共同淨利 × 80% ÷ 8) + 非共同部分個人份額
- 共同基金 = 共同淨利 × 20%

**計算鏈影響路徑**：交易寫入 → 自動稅重算 → 共同支出計算 → 共同淨利 → 成員分配 / 基金提撥 → 基金餘額（扣除 paidByFund）

## Goals / Non-Goals

**Goals:**

- 每個 show 的收入自動產生對應稅務預留支出，數字永遠跟隨收入變動
- 稅務預留進入既有 `excludedMembers` 分配邏輯，不影響成員分配的正確性
- 基金支付機制讓成員分配淨利「不被基金支付的費用影響」、基金餘額可明確看到
- 年度會計費 $13,000 以一筆交易記錄，歸在「共同基金支出」虛擬 show
- 所有現有資料可透過一次性 migration 補齊到新模型

**Non-Goals:**

- 不做重複交易（每月自動產生）機制
- 不做稅率 UI 設定介面（修改常數即可）
- 不處理過去已由基金支付但未標記的歷史資料（確認無此情況）
- 不改動 Opentix 匯入、結清狀態、墊款人等既有流程

## Decisions

### Decision: 稅務預留採「per show × per excludedMembers 分組」一筆自動支出（A2 方案）

**選擇**：每個 show 下，依 `excludedMembers` 字串值 group by，每組聚合收入總和後產生一筆 `autoGenerated=true` 的「稅務預留」支出。金額 = 組內收入總和 × `TAX_RESERVE_RATE`。

**替代方案**：
- A1（每個 show 固定一筆、excludedMembers 空）：遇到 show 內有混合分配的收入時會把稅分攤給根本沒分到收入的成員，數字錯誤。
- A3（每筆收入對應一筆稅支出）：交易數量膨脹 2 倍，更新成本高，無法「改金額不增筆數」。

**理由**：A2 在單純案例（全員共同）結果跟 A1 一樣乾淨；混合案例也不會算錯；資料結構沿用既有 `excludedMembers` 語義，分析頁零改動。

### Decision: 自動稅支出的重算採「整組刪除後重建」

**選擇**：任何交易寫入操作影響到 show X 時，GAS 後端先刪除 show X 所有 `autoGenerated=true AND category='稅務預留'` 的交易，再依當前收入重新計算並插入。

**替代方案**：
- Incremental update（找對應 excludedMembers 組的稅支出加減）：邏輯複雜，edge case 多（刪除最後一筆收入時要刪掉對應稅支出）
- Scheduled daily recalc：會有一段時間數字錯

**理由**：刪除重建的邏輯最單純、無狀態，稅支出筆數遠少於收入筆數，效能不是問題。

### Decision: 新增 `paidByFund: boolean` 欄位取代「用 category 判斷」

**選擇**：在交易結構新增獨立的 boolean 欄位 `paidByFund`。

**替代方案**：
- 新增「共同基金支出」category：限制太死，基金可能支付任何類型的支出（行政雜支、平台手續、場地等）
- 用虛擬成員「共同基金」放 `excludedMembers`：語義混亂，基金不是成員

**理由**：flag 與 category 解耦，基金可以支付任何類別；與 `excludedMembers` 也正交，規則清晰。

### Decision: paidByFund 支出不計入共同支出

**選擇**：`paidByFund=true` 的支出從「共同支出」聚合中完全排除（即使 `excludedMembers === ''`）。基金餘額 = 累積提撥 − Σ(paidByFund 支出)。

**替代方案**：
- paidByFund 算共同支出，再從基金扣一次：會雙重扣除（先降共同淨利 → 再降基金）
- paidByFund 算共同支出但基金從「共同支出」扣：會讓成員分配淨利被錯誤影響

**理由**：讓「成員可分配淨利」不受基金支付費用影響是此機制的核心訴求，隔離是唯一乾淨解。

### Decision: 新增「共同基金支出」作為虛擬 show（B2）

**選擇**：在 default show list 新增「共同基金支出」。此 show 下的支出 `paidByFund` 預設 true。

**替代方案**：
- 歸到「會員與其他收支」：該 show 損益會被基金支出扭曲
- 允許 show 為空：既有 UI / 報表全部要處理 null show，改動大

**理由**：維持「每筆交易都有 show」的既有不變量，專案損益表一眼看出基金支出規模，邏輯閉合。

### Decision: paidByFund 支出的 excludedMembers 強制空

**選擇**：UI 勾選「由共同基金支付」時隱藏 excludedMembers checkbox 區塊；後端寫入時強制 `excludedMembers = ''`。

**替代方案**：
- 允許 paidByFund + excludedMembers 共存：語義矛盾（基金是全員共有），且不會影響計算（反正不進分配）

**理由**：資料層避免不必要的 noise；UI 層移除無意義的輸入。

### Decision: 稅率採硬編碼常數

**選擇**：在 `gas/Code.gs` 頂部宣告 `const TAX_RESERVE_RATE = 0.03;`。前端如需顯示稅率，透過 `getAnalytics` 回傳。

**替代方案**：
- 設定 sheet：可動態改，但目前沒有系統設定頁，為了這一個值不值得
- hardcode 分散在多處：之後改率值會漏

**理由**：集中一處、未來調整改一行；不過早設計 UI。

### Decision: 年度會計費採「手動建一筆年度交易」而非月度自動

**選擇**：migration 時建立一筆 $13,000 的支出（date=當年任一日期、show=共同基金支出、category=行政雜支、paidByFund=true、notes=「年度會計費用」）。

**替代方案**：
- 月度自動產生（12 筆 $1,083）：需要整套重複交易機制，目前僅此一需求，YAGNI

**理由**：1 筆 vs 12 筆在分析上沒有差別，實作成本差一個數量級。

## Risks / Trade-offs

- [**風險：自動稅重算邏輯 bug 會污染歷史資料**] → Mitigation：migration 前先 dump Google Sheet 備份；重算函式加入單元測試（如 GAS 允許）或先在測試 sheet 驗證。
- [**風險：使用者手動編輯自動稅支出造成數字不一致**] → Mitigation：UI 層禁用編輯 / 刪除按鈕；GAS 後端 validation 拒絕對 `autoGenerated=true` 的更新（除非 recorder 為系統）。
- [**風險：既有交易可能沒有 `excludedMembers` 欄位（legacy 資料）**] → Mitigation：讀取時 normalise 為空字串；migration 跑之前先 audit，缺失欄位補空字串。
- [**權衡：稅率改動需改 code**] → 接受；使用者確認目前只有一個稅率，未來有需要再做設定頁。
- [**風險：「共同基金支出」show 出現在 checklist 頁面可能造成混淆**] → Mitigation：checklist 頁面可過濾排除虛擬 show，或明顯標示；此細節移到 tasks 處理。
- [**權衡：paidByFund 支出不進共同支出 → 支出分類佔比可能「少算了基金付掉的部分」**] → 接受；分析頁在基金區塊已清楚顯示基金支出總額，重複顯示反而混亂。

## Migration Plan

1. **備份**：migration 前 export 整份「收支紀錄」sheet 為 CSV 存檔。
2. **加欄位**：在 Google Sheet 擴增 `paidByFund`、`autoGenerated` 兩個欄位，現有列填 false。
3. **加虛擬 show**：在「演出清單」sheet 新增「共同基金支出」。
4. **回填自動稅支出**：掃描所有既有 show，依 `excludedMembers` 分組、計算 × 0.03、寫入對應 `autoGenerated=true` 的「稅務預留」支出列。
5. **建立年度會計費交易**：寫入一筆 $13,000 支出到「共同基金支出」show，paidByFund=true。
6. **部署新前端 / GAS**：上線後首次 `addTransaction` 時自動稅重算邏輯已生效，與 migration 結果一致。
7. **驗證**：比對 migration 前後共同淨利、成員年度分配淨利的差異符合預期（差值應為共同稅務預留對應的 20% / 80% 分配）。

**Rollback**：restore CSV 備份 → redeploy 舊版 GAS / 前端。因為 migration 只新增欄位 / 新增列，不修改既有列資料，rollback 相對乾淨。
