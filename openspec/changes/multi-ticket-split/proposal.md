## Why

目前應援匯入的訂單編號精確配對有兩個問題：

1. **合購訂單無法拆分**：一筆訂單可能包含多場活動的票（如 `5/30 B3 內野應援票 x 1  7/11 B3 資深經理席 x 1`），撥款明細只有一筆合計金額，無法正確分拆到各活動
2. **同名票券跨活動**：「高腳圓桌 & 包廂對號票」在 4 月號和 5 月號完全同名同價（$550），系統無法自動判斷歸屬哪場活動

這兩個問題會導致分潤計算錯誤，因為金額被歸到錯誤的活動或無法拆分。

## What Changes

- **新增第三上傳區「活動報名狀態」**：接受 `.xlsx` 格式，從中建立票價表（票券全名 → 單價）和活動歸屬表（票券全名 → 活動名稱，從檔名提取）
- **拆分多票合購訂單**：解析 `票券名稱及數量` 欄位中的 `AAA x N  BBB x M` 格式，將一筆撥款拆成多張獨立票券。每張票的金額從票價表查出精確單價，手續費按票價比例分攤
- **活動自動歸屬**：拆分後的每張票，透過活動報名狀態的票券全名（含座位號）查找所屬活動，自動歸入對應專案
- **同名同價票的處理**：當同一票券名稱出現在多個活動中且價格相同時，使用訂單建立時間對比各活動的售票起訖日推斷歸屬；若仍無法判斷，在 mapping UI 標記讓使用者手動確認

## Non-Goals

- 不改動撥款明細（左欄）和票券/周邊訂單（右欄）的既有上傳與配對邏輯
- 活動報名狀態為可選上傳 — 不上傳時系統退回既有行為（groupName 批次對應）
- 不處理周邊訂單的拆分（周邊訂單沒有跨活動合購的問題）

## Capabilities

### New Capabilities

（無）

### Modified Capabilities

- `oen-cashflow-import`: 新增活動報名狀態上傳區作為票價表與活動歸屬參考源；新增多票合購拆分邏輯；修改 mapping 流程支援活動自動歸屬與同名票衝突處理

## Impact

- 受影響的 spec：`oen-cashflow-import`
- 受影響的程式碼：
  - `import.html`：新增第三上傳區 UI
  - `js/import.js`：票價表建立、多票拆分邏輯、活動歸屬邏輯、mapping UI 更新
